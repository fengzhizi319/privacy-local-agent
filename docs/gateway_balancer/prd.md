# 代理转发与负载均衡网关 (Gateway / Load Balancer) PRD

## 1. 业务背景与痛点
隐私计算本地代理 `privacy-local-agent` 目前仅支持单实例部署，同时监听 REST (FastAPI) 与 gRPC 接口。在面临大规模并发请求时（例如：高吞吐量的数据脱敏、高频的差分隐私计算、大规模数据分类任务），单实例受限于 Python 解释器的 GIL 限制以及单机的 CPU/内存 瓶颈，容易出现请求积压、延迟增加甚至服务不可用的情况。

为了解决这一痛点，参考隐语生态后端项目 `SecretPad`（基于 Kuscia 架构，通过 Envoy 代理与网关实现高可用和跨域请求路由）的设计理念，为 `privacy-local-agent` 引入统一的**代理转发与负载均衡网关 (API Gateway & Load Balancer)**。该网关将作为统一的请求入口，自动、动态地将大流量的 REST 接口与 gRPC 请求分发到后端多个健康的 `privacy-local-agent` 工作节点，从而实现水平扩展（Scale-out）与高可用性（High Availability）。

## 2. 产品定位与目标
- **定位**：作为一个轻量级、高性能的独立组件（也可与 Agent 运行于同镜像中，仅启动参数不同），支持双协议（HTTP/REST 与 gRPC）流量代理，透明化后端节点集群。
- **目标**：
  - 支持横向扩展，能够弹性挂载任意数量的后端工作实例。
  - 提供多策略的负载均衡，有效应对请求尖峰，提高整体系统的吞吐量（QPS）。
  - 实现故障隔离与自动恢复（故障检测与自动剔除/上线），提供高可用的服务保障。

## 3. 用户场景与用例 (Use Cases)
- **场景一：高吞吐数据脱敏 (Data Masking)**
  - 用户需要对上百万条敏感记录（手机号、身份证、姓名）进行脱敏处理。网关接收并发的批量脱敏请求，利用轮询或最小连接数策略，将请求均分给后端 5 个处理节点，响应时间降低至单实例的 1/5。
- **场景二：差分隐私高频聚合 (DP Aggregation)**
  - 多个客户端高频发起差分隐私计数/均值计算。由于隐私预算由具体命名空间维护，网关在转发时能保证各节点预算策略一致，并合理分散计算负载。
- **场景三：节点故障自动容错 (Failover)**
  - 后端某一个工作节点由于内存溢出（OOM）崩溃或网络中断。网关内置的健康检查机制能在数秒内感知，将其从可用池中剔除，并自动将后续流量分发至剩余健康节点，保证上层调用不受影响。

## 4. 功能性需求 (Functional Requirements)
### 4.1 协议转发与代理能力
- **REST (HTTP) 转发**：
  - 网关必须支持对 `/v1/privacy/*` 以及 `/health` 等所有 HTTP 端点的透明代理。
  - 采用通用路径匹配转发，避免硬编码接口路径，以便对未来新增接口自动适配。
  - 必须正确透传请求头、查询参数（Params）及 Body，且返回时透传后端的响应头和状态码。
- **gRPC 转发**：
  - 网关必须实现 `PrivacyService` 服务（包含 `Mask`, `MaskRecord`, `Hash`, `DPCount`, `DPSum`, `DPMean`, `KAnonymizeRecord`, `ObfuscateQuery`, `ClassifyField`, `ClassifyRecord`, `ClassifyTable`, `Health` 等所有 RPC 方法）。
  - 使用 gRPC 客户端连接池，将接收到的 protobuf 请求动态转发至选定的后端工作节点，并将响应返回给客户端。

### 4.2 负载均衡策略
网关必须支持以下三种核心负载均衡算法，并允许通过配置进行切换：
1. **轮询 (Round-Robin)**：请求按顺序分配给后端每个可用节点。
2. **随机选择 (Random)**：在所有健康的后端节点中随机选择一个分发。
3. **最小连接数 (Least Connections)**：优先将请求分发给当前处于活跃处理状态连接数最少的节点，适用于执行时长差异较大的请求（如大表分类与单字段脱敏混合场景）。

### 4.3 节点健康检查与自愈
- **主动健康检查**：
  - 网关定时（例如每 5 秒）向所有配置 of 后端节点发送健康检查请求。
  - 对 REST 端点调用 `/health` 接口，对 gRPC 端点调用 `Health` 接口。
  - 必须两者都返回正常（HTTP 200，gRPC 返回 `"status": "ok"`）才视为节点健康。
- **动态状态维护**：
  - 当发现节点状态改变（健康 -> 异常 或 异常 -> 健康），网关应实时更新可用节点池。
  - 当可用节点池为空时，网关对于 incoming 请求需返回特定的不可用错误（HTTP 503，gRPC UNAVAILABLE）。

### 4.4 配置灵活性
- 支持通过 YAML 配置文件或环境变量配置以下参数：
  - 网关监听的 REST 端口与 gRPC 端口。
  - 后端节点列表（每个节点需配置 `http_url` 与 `grpc_address`）。
  - 负载均衡策略 (`round_robin` / `random` / `least_connections`)。
  - 健康检查间隔时间（秒）、超时时间等。

### 4.5 分布式共享预算一致性 (Distributed Shared Budget Consistency)
- 在多实例负载均衡模式下，各节点内存是隔离的。为防止多次计算越过隐私预算限制，必须支持共享持久化预算账本。
- 使用无外部依赖的文件锁（如 SQLite 内置事务排他锁）支持多节点进程共享同一账本文件，保证 epsilon/delta 的扣减是全局原子的，且超出限制时抛出异常。
- 提供向下兼容性：未配置数据库时，默认回退为单机内存模式运行。

## 5. 非功能性需求 (Non-Functional Requirements)
- **高性能与低延迟**：
  - 网关转发引入的额外耗时需限制在 5ms 以内。
  - REST 部分必须基于 FastAPI 的异步机制，gRPC 部分必须基于 `grpc.aio`（AsyncIO）实现，确保高并发网络 I/O 下网关自身不会成为瓶颈。
  - **连接池复用与长连接**：网关必须使用应用级单例 HTTP 客户端连接池以复用连接，避免在高并发场景下因频繁握手或套接字耗尽（OSError）导致服务失败。
- **鲁棒性与异常处理**：
  - 必须优雅处理各种连接超时、重试逻辑、连接被拒绝等底层异常，并返回客户端友好的提示，严禁服务进程崩溃。
- **代码结构一致性**：
  - 保持与现有 `privacy-local-agent` 良好的模块隔离，新增代码应在专用模块中，减少对原单机业务逻辑的侵入。
